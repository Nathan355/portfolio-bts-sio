C1 - Recenser et identifier les ressources numériques

Gestion de parc et inventaire automatisé d'équipements et de logiciels avec GLPI.

🎯 Objectif

Créer et maintenir un système d'inventaire centralisé et automatisé afin d'assurer une visibilité complète sur le parc machine et applicatif de l'organisation.

📋 Procédure de mise en œuvre

Étape 1 : Déploiement de l'infrastructure serveur (LAMP)

Nous installons l'environnement serveur requis pour héberger l'application web GLPI.

  1. Mettre à jour les dépôts système : sudo apt update && sudo apt upgrade
  2. Installer la pile applicative : sudo apt install apache2 mariadb-server php php-mysql php-gd php-mbstring php-xml php-curl
  3. Pourquoi ? GLPI nécessite un serveur HTTP (Apache) pour diffuser l'interface, un moteur d'exécution (PHP) pour traiter la logique logicielle, et un SGBD (MariaDB/MySQL) pour stocker la base de données relationnelle du parc.
  4. Télécharger et extraire GLPI dans le répertoire web : /var/www/html/glpi
  5. Attribuer les droits à Apache : sudo chown -R www-data:www-data /var/www/html/glpi
Étape 2 : Déploiement et liaison des agents d'inventaire

Afin d'automatiser la remontée des composants matériels et logiciels sans intervention humaine répétitive.

  • Sur un environnement Windows : Exécuter l'installateur d'agent .msi. Lors de l'assistant, spécifier l'adresse de destination cible : http://[IP_SERVEUR]/glpi/plugins/glpiinventory/.
  • Sur un environnement Linux (Debian/Ubuntu) :
    Installation de l'agent : sudo apt install glpi-agent
    Configuration de l'adresse du serveur dans /etc/glpi-agent/agent.cfg : server=http://[IP_SERVEUR]/glpi/
    Forcer la première remontée immédiate : sudo glpi-agent --force
Étape 3 : Contrôle et exploitation des données

Validation de la bonne réception des métadonnées systèmes.

  1. Se connecter à la console d'administration GLPI.
  2. Naviguer vers le menu Parc > Ordinateurs.
  3. Vérifier l'exactitude des fiches générées dynamiquement (capacités physiques de la RAM, architecture CPU, partitions de stockage et licences de logiciels détectées).

✅ Critères de Validation du Jury

Présentation d'un serveur fonctionnel contenant au moins 3 machines distinctes inventoriées automatiquement avec le listing complet de leurs dépendances matérielles et logicielles.

C7 - Collecter, suivre et orienter les demandes

Configuration d'un centre de services (Helpdesk) et cycle de vie d'un incident.

🎯 Objectif

Centraliser, qualifier et orchestrer la résolution des incidents techniques et des demandes de services afin de respecter les engagements de support.

📋 Procédure de mise en œuvre

Étape 1 : Structuration de l'organisation (Entités)
  1. Se rendre dans Administration > Entités.
  2. Créer le cloisonnement organisationnel de la structure (ex: Direction, RH, Comptabilité, Technique).
  3. Pourquoi ? Le découpage en entités garantit l'isolation des données, la gestion fine des droits d'accès des techniciens et le ciblage précis des utilisateurs.
Étape 2 : Cycle de vie nominal d'un ticket

Suivi rigoureux du flux de traitement de l'anomalie jusqu'à sa résolution :

  • Ouverture (Nouveau) : L'utilisateur final se connecte à son portail simplifié et soumet son problème (ex : dysfonctionnement d'imprimante).
  • Prise en charge (En cours) : Le technicien qualifie la demande, modifie le statut en "En cours", définit la priorité de l'incident et consigne les premières actions dans l'historique de suivi.
  • Clôture (Clos) : La solution technique est appliquée. Le ticket passe à l'état "Résolu" puis "Clos" après validation ou absence de contestation du demandeur.
Étape 3 : Pilotage et démarche d'amélioration continue
  1. Consulter le tableau de bord décisionnel via Assistance > Statistiques.
  2. Analyser les indicateurs de performance (KPI) : temps moyen de prise en charge, récurrence des pannes.
  3. Générer des articles de Base de Connaissances (FAQ) à partir des résolutions fréquentes pour rendre l'utilisateur autonome et réduire le volume global de tickets entrants.

✅ Critères de Validation du Jury

Démonstration d'un historique d'incidents clôturés avec catégorisation des demandes, affectation à des groupes de techniciens dédiés et extraction de graphiques de performance.

C11 - Référencement des services en ligne et mesure de visibilité

Déploiement d'une infrastructure Web sur GitHub Pages et intégration d'un tracking asynchrone Google Analytics.

🎯 Objectif

Assurer le déploiement continu d'un livrable web accessible publiquement et configurer une solution de collecte d'audience pour mesurer les indicateurs clés de performance (KPI).

📋 Procédure de mise en œuvre

Étape 1 : Hébergement et publication via GitHub Pages
  1. Créer un dépôt (Repository) public sur GitHub et y pousser l'arborescence des fichiers source du projet (HTML, CSS, JS).
  2. Accéder à l'onglet Settings du dépôt, puis section Pages.
  3. Configurer la source de construction sur la branche principale main ou master et valider.
  4. Pourquoi ? GitHub Pages fournit une infrastructure d'hébergement statique automatisée, sécurisée par un certificat SSL (HTTPS) natif et synchronisée directement avec le gestionnaire de versions Git.
Étape 2 : Implémentation de la solution d'analyse d'audience
  1. Initialiser une propriété Web au sein de la console Google Analytics afin de récupérer l'identifiant unique de mesure (G-XXXXXXXXXX).
  2. Insérer la balise de suivi JavaScript fournie au sein de l'en-tête globale du site :
<!-- Global site tag (gtag.js) - Google Analytics -->
<script async src="https://www.googletagmanager.com/gtag/js?id=G-XXXXXXXXXX"></script>
<script>
  window.dataLayer = window.dataLayer || [];
  function gtag(){dataLayer.push(arguments);}
  gtag('js', new Date());
  gtag('config', 'G-XXXXXXXXXX');
</script>
Étape 3 : Audit de la télémétrie
  • Naviguer sur le site web nouvellement publié pour générer du trafic.
  • Ouvrir le tableau de bord Google Analytics dans l'onglet Temps réel.
  • Vérifier la bonne capture de l'événement de visite (comptage de l'utilisateur actif, localisation géographique et terminaux utilisés).

✅ Critères de Validation du Jury

Vérification de l'accès public au site web via son URL d'hébergement sécurisée et démonstration en direct de la capture des flux utilisateurs dans l'outil d'analyse.

C14 - Planification des activités

Gestion de projet et industrialisation : Application de Suivi des Certifications pour Open Value.

🎯 Objectif

Orchestrer le cycle de vie du développement d'une application métier critique de suivi des compétences et certifications, de sa modélisation à sa sécurisation applicative.

📋 Architecture et Spécifications Logicielles

Structure Technique de l'Application
  • Backend : Python s'appuyant sur le micro-framework Flask pour gérer le routage.
  • Persistance des données : Modélisation relationnelle exploitant SQLite via l'ORM SQLAlchemy pour s'abstraire des requêtes SQL brutes et sécuriser les transactions.
  • Pattern Architectural : Architecture MVC (Modèle-Vue-Contrôleur) séparant strictement la logique de données de la couche d'affichage.
Fonctionnalités Clés Implémentées
  1. Gestion des ressources : Opérations CRUD complètes (Create, Read, Update, Delete) sur les profils des collaborateurs et le catalogue de certifications.
  2. Ordonnancement et alertes : Algorithme d'analyse temporelle programmé pour calculer les dates de validité et lever des alertes automatiques à J-7 avant expiration des compétences.
Mécanismes de Sécurisation Intégrés (Audit Qualité)

Afin de prémunir l'application métier contre les vulnérabilités majeures du Top 10 OWASP :

  • Injections SQL : Utilisation systématique de requêtes paramétrées via l'ORM.
  • Hachage des secrets : Implémentation de l'algorithme robuste SHA256 pour le stockage des mots de passe.
  • Failles CSRF et Sessions : Injection de jetons d'authentification uniques au sein des formulaires et protection des cookies applicatifs.

✅ Critères de Validation du Jury

Présentation du code source structuré, démonstration des alertes de planification automatique et passage réussi d'un test d'injection de données malveillantes.

C17 - Déploiement d’un service

Conteneurisation d'applications et isolation d'environnements de production à l'aide de Docker.

🎯 Objectif

Industrialiser le déploiement d'un service applicatif en encapsulant ses composants et ses dépendances logicielles au sein d'un conteneur isolé et immuable, garantissant un comportement identique sur tout environnement.

📋 Procédure de mise en œuvre

Étape 1 : Conception du descripteur d'image (Dockerfile)

Création d'un fichier texte nommé Dockerfile à la racine du projet définissant les étapes de construction de notre environnement :

# Définition de l'image de base (Serveur web léger de production)
FROM nginx:latest

# Copie des artefacts statiques du projet vers le répertoire de diffusion du serveur
COPY ./mon-site /usr/share/nginx/html

# Exposition du port d'écoute réseau standard
EXPOSE 80

Pourquoi ? Cette méthode élimine le syndrome du "ça marche sur ma machine" en figeant les versions des packages système et du serveur web indépendamment du système d'exploitation hôte.

Étape 2 : Build de l'image Docker

Compilation de notre environnement isolé :

docker build -t mon-app-web:1.0 .

Cette commande instruit le démon Docker de lire le Dockerfile du répertoire courant (.) et d'empaqueter l'application sous l'étiquette (tag) spécifiée.

Étape 3 : Instanciation et exécution du conteneur

Mise en production du service avec redirection des flux réseaux :

docker run -d -p 8080:80 --name conteneur-production mon-app-web:1.0
  • -d : Exécute le conteneur en tâche de fond (mode détaché).
  • -p 8080:80 : Mappe le port 8080 de la machine hôte vers le port 80 interne du conteneur.

✅ Critères de Validation du Jury

Exécution de la commande docker ps montrant le conteneur actif et accès réussi à l'application depuis un navigateur externe via l'adresse http://[IP_HOTE]:8080.
